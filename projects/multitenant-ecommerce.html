Learning Project: Built by following a structured tutorial to explore advanced Next.js patterns, multi-tenant architecture, and headless CMS integration. The focus was understanding how production-grade SaaS platforms are structured.

Project Overview

A multi-tenant e-commerce SaaS platform where each seller gets their own subdomain-isolated storefront. The platform is powered by PayloadCMS as the headless CMS and admin backend, Next.js for server-rendered storefronts, and Stripe Connect for multi-party payment processing.

  • Multi-Tenancy: Each tenant operates on their own subdomain (e.g. store.funroad.com), with isolated products and settings managed through role-based access
  • CMS-Driven Backend: PayloadCMS provides a full admin panel for content, product, and user management with built-in multi-tenant plugin support
  • Type-Safe API Layer: tRPC connects Next.js frontend to backend procedures fully end-to-end typed — no REST guesswork
  • Payment Infrastructure: Stripe Connect allows individual tenants to onboard and receive payments directly into their own Stripe accounts
  • Product Discovery: Category and tag-based filtering with pagination for efficient product browsing across stores

Key Features

Subdomain-Based Multi-Tenancy

Each tenant store runs on a unique subdomain. Next.js middleware resolves the subdomain at the edge and scopes all data queries to that tenant automatically.

Role-Based Access Control

Super-admins have global access across all tenants. Tenant admins can only manage their own store's products and settings — enforced at the CMS level.

PayloadCMS Authentication

Full authentication flow using Payload's built-in JWT-based auth — registration, login, session management, and protected collection access.

Stripe Connect

Tenants connect their Stripe accounts to receive payments directly. The platform handles onboarding status and prevents product creation until Stripe details are submitted.

tRPC Type-Safe API

All data-fetching between the Next.js app and backend goes through tRPC routers — providing compile-time type checking and eliminating runtime type mismatches.

Categories & Tags

Products are organised with category and tag relationships in PayloadCMS, enabling multi-faceted filtering on the storefront.

Filters & Pagination

Product listings support category/tag filtering and paginated results, keeping storefronts performant regardless of catalogue size.

PayloadCMS Admin Panel

Fully featured admin interface auto-generated by Payload — manage products, tenants, users, media, and refund policies without building a separate dashboard.

Architecture

Multi-Tenant Subdomain Routing

Next.js middleware intercepts every incoming request and reads the subdomain from the host header. The subdomain slug is matched against the tenants collection in PayloadCMS, and all downstream data queries (products, categories, etc.) are automatically scoped to that tenant. This means a single deployed application serves any number of stores without separate deployments.

PayloadCMS + Multi-Tenant Plugin

PayloadCMS's multiTenantPlugin attaches a tenant relationship to specified collections (products in this case). Access control hooks ensure that a logged-in tenant admin can only read and write records belonging to their own tenant, while super-admins bypass these restrictions. This is configured declaratively — no manual access control logic needed per collection.

Stripe Connect Onboarding Flow

When a tenant is created, a Stripe Connect account is provisioned and the stripeAccountId is stored on the tenant record. The stripeDetailsSubmitted flag is flipped once Stripe confirms the onboarding is complete. Product creation is blocked in PayloadCMS if this flag is false, ensuring no tenant can list products before they can receive payments.

tRPC Data Layer

The frontend communicates with PayloadCMS data through tRPC routers (e.g. products, categories, tenants). Each router procedure is typed end-to-end using Payload's auto-generated TypeScript types, so the frontend always knows exactly what shape of data to expect. React Query handles caching, refetching, and loading states transparently.
