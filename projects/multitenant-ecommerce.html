Project Overview

A multi-tenant e-commerce SaaS platform where each seller's data and products are isolated through role-based access control. Powered by PayloadCMS as the headless CMS and admin backend, Next.js 16 for server-rendered storefronts, and tRPC for end-to-end type-safe data access.

  • Multi-Tenancy: Each tenant's products and settings are isolated through PayloadCMS role-based access control, with tenant admins scoped to their own data and super-admins having global access
  • CMS-Driven Backend: PayloadCMS provides a full admin panel for content, product, and user management with built-in multi-tenant plugin support
  • Type-Safe API Layer: tRPC connects Next.js frontend to backend procedures fully end-to-end typed, with no REST guesswork
  • Secure Auth: httpOnly cookie-based auth set server-side; the JWT never touches localStorage, preventing token theft via XSS
  • Product Discovery: Hierarchical category nav, tag filtering, price range, and infinite scroll product listings with server-side prefetching
  • Product Library: Authenticated users can save and revisit purchased products from their personal library
  • Database Seeding: Seed script to populate tenants, products, and categories for development and demo environments

Key Features

Role-Based Multi-Tenancy

Each tenant's data is isolated at the database level via PayloadCMS access control hooks. Tenant admins can only view and manage their own products and settings; super-admins have unrestricted access across all tenants.

Role-Based Access Control

Super-admins have global access across all tenants. Tenant admins can only manage their own store's products and settings, enforced at the CMS level.

Secure Cookie-Based Auth

Payload's built-in auth writes the JWT into an httpOnly cookie server-side, never exposed to JavaScript, protecting against XSS token theft. Session state is refreshed client-side via TanStack Query invalidation.

Infinite Scroll

Product listings use TanStack Query's useInfiniteQuery with server-side prefetching; the first page is rendered on the server and subsequent pages load seamlessly as the user scrolls.

tRPC Type-Safe API

All data-fetching between the Next.js app and backend goes through tRPC routers, providing compile-time type checking and eliminating runtime type mismatches.

Hierarchical Categories & Tags

Categories support parent/subcategory nesting; a category can reference itself as a parent and auto-populates its children. Combined with flat tags, this powers a rich multi-level nav and filter UI.

Filters & Pagination

Product listings support category/tag filtering and paginated results, keeping storefronts performant regardless of catalogue size.

PayloadCMS Admin Panel

Fully featured admin interface auto-generated by Payload for managing products, tenants, users, media, and refund policies without building a separate dashboard.

Product Library

Authenticated users have a personal product library where they can access and revisit their purchased products, persisted in the database and protected behind auth.

Rich Text Descriptions

Product descriptions use Payload's Lexical rich text editor. Sellers can format content with headings, lists, and inline styles, stored as structured JSON.

Architecture

Tenant Data Isolation

The multiTenantPlugin attaches a tenant relationship to the products collection. When a tenant admin logs in, PayloadCMS access control hooks filter all queries to only return records belonging to their tenant. Super-admins bypass these restrictions and can manage any tenant's data. This is configured declaratively, with no manual access control logic needed per collection.

PayloadCMS + Multi-Tenant Plugin

PayloadCMS's multiTenantPlugin attaches a tenant relationship to specified collections (products in this case). Access control hooks ensure that a logged-in tenant admin can only read and write records belonging to their own tenant, while super-admins bypass these restrictions. This is configured declaratively, with no manual access control logic needed per collection.

Authentication Flow

The user submits credentials via the sign-in form. The tRPC login mutation calls Payload's local auth API server-side, which validates credentials and returns a JWT. That token is written into an httpOnly cookie; the browser sends it automatically on every request, but JavaScript cannot read it, preventing XSS-based token theft. On the client, queryClient.invalidateQueries refreshes the session state after login or logout.

tRPC + TanStack Query Data Layer

The frontend communicates with PayloadCMS through tRPC routers (products, categories, tenants). Each procedure is typed end-to-end using Payload's auto-generated TypeScript types. TanStack Query 5 handles caching, background refetching, and infinite scroll via useInfiniteQuery; the first page is server-prefetched for fast initial load, with subsequent pages fetched on demand.

What I Learned

Multi-Tenant Patterns

How to design a single codebase that isolates data per tenant using scoped database queries and declarative access control.

Headless CMS Architecture

How PayloadCMS acts as both a backend API and admin interface, replacing traditional custom-built admin dashboards.

End-to-End Type Safety

Using tRPC alongside Payload's auto-generated TypeScript types to ensure frontend and backend are always in sync at compile time, with no manual type maintenance.

Schema Validation with Zod

React Hook Form + Zod handles all form state and input validation. Schemas are defined once and shared between the form and the tRPC input validators.

Next.js App Router

Leveraging server components, route groups, and middleware for routing logic and server-side data prefetching.

Access Control at Scale

Declarative role-based access in PayloadCMS, separating super-admin and tenant-scoped permissions without duplicating logic across collections.

Bun Runtime

Used Bun as the JavaScript runtime and package manager, which provides significantly faster installs and dev server startup compared to Node.js/npm.

Development Workflow

Feature Branching

Each feature, fix, or logical unit of work was developed on its own dedicated branch, keeping main stable throughout development. This mirrors standard team workflows where changes are isolated until they're reviewed and ready to merge.

Pull Requests & Code Review with CodeRabbit

Every feature branch was merged via a pull request rather than pushed directly to main. CodeRabbit was integrated as an automated AI code reviewer on each PR. It generated descriptive summaries of the changes, flagged potential bugs, highlighted code quality issues, and suggested improvements before the code was merged. This enforced a review step on every change and built the habit of treating each PR as a documented unit of work.

  • Automated summaries: CodeRabbit produced a plain-English walkthrough of every PR covering what changed, why, and what to watch out for
  • Inline feedback: Issues were raised as review comments on specific lines, matching the experience of a human peer review
  • Error detection: Caught logic issues, missing edge cases, and code smells before they landed on main
  • Clean commit history: Merging via PRs kept the Git history readable and each change traceable to a purpose
