Project Overview

FriendChise is a multi-tenant operations management platform designed for franchise businesses. It handles task scheduling, timetable templates, role-based access control, and team coordination across multiple branches.

The Problem: Small-to-medium franchise operations (food chains, cleaning services, retail) rely on group chats, spreadsheets, and verbal instructions to manage recurring tasks. There's no single system where an owner can define what needs to happen, a manager can schedule who does it, and a worker can see their daily checklist — all with proper access control.

The Solution: A web app where franchise owners define task libraries and timetable templates at the franchise level, managers apply those templates to generate weekly schedules for their branch, and workers check off tasks from a daily view. Everything is scoped by organization with role-based permissions.

Key Features (MoSCoW Prioritized)

Multi-Tenant Organizations

Create/join organizations. Each org is a completely separate data silo with isolated users, tasks, and schedules.

Role-Based Access Control

Owner, Manager, Worker roles with granular permissions. Each role has specific capabilities defined by OrgPermission enums.

Task Library

Create reusable task templates with title, duration, description, and eligible roles. Tasks can be assigned to schedules.

Timetable Templates

Define repeating weekly patterns using relative days (Day 1–7). Drag/drop task placement in the template editor.

Materialised Scheduling

Apply a template to generate real TaskInstance rows with absolute dates. Each instance can be independently edited.

Timetable View

Calendar (week/day) and simple (list) modes. Status indicators, week navigation, and overlap detection.

Task Instance Management

Assign members, update status (TODO → IN_PROGRESS → DONE/SKIPPED). Full audit trail of who completed what.

Member Management

Invite by email, assign roles, remove members. Org-scoped permissions control who can manage members.

Architecture & Design Decisions

Materialised Scheduling vs Virtual Projection

Chose: Materialised scheduling. When a template is applied to a week, real TaskInstance rows are generated with absolute dates. This allows each instance to be independently edited (reassign a specific task, change status, adjust time) without a complex exception layer. Virtual projection can't support per-instance overrides. The storage cost is minimal — a few hundred rows per week per org.

Auth Config Split for Edge Runtime

Next.js middleware runs on the Edge runtime, which cannot import Node.js modules like @prisma/client. Auth.js config is split into two files:

  • auth.config.ts — Edge-compatible config (no Prisma). Used by middleware for fast auth checks.
  • auth.ts — Full config with Prisma adapter and JWT session callback. Used by API routes and server components.

Service Layer Pattern

Both Server Actions (for web UI) and API Routes (for external/mobile clients) are thin wrappers that handle auth and input validation, then delegate to lib/services/. The service layer holds all business logic and is shared between both paths. Server Actions additionally call revalidatePath to invalidate the Next.js cache.

Discriminated Union Result Types

Authorization helpers (requireUser, requireOrgMember, requireOrgPermission) return { ok: true, user, membership } or { ok: false, response: NextResponse(401|403) }. Routes can early-return with if (!authz.ok) return authz.response — clean, consistent, no exceptions for control flow.

Data Model

Key models in the PostgreSQL schema:

  • Organization — Top-level tenant. Owns all other resources.
  • User / Account / Session — Auth.js managed. OAuth account linking.
  • Membership — Join table: User ↔ Organization, with optional Role.
  • Role / RolePermission — Org-scoped roles with permission sets.
  • Task — Reusable task template (title, duration, description, peopleRequired).
  • TaskEligibility — Links Task to eligible Roles.
  • TimetableTemplate — Named cycle profile with cycleDays count.
  • TaskInstance — Dual-purpose: template placements (relative dayOffset) OR materialised entries (absolute scheduledStartAt).
  • TaskInstanceAssignee — Many-to-many: TaskInstance ↔ Membership.

REST API Endpoints

  • POST /api/orgs — Create org (auto-creates Owner/Member roles)
  • GET/POST/DELETE /api/orgs/[orgId]/memberships — Manage members
  • GET/POST/DELETE /api/orgs/[orgId]/tasks — Task templates
  • GET/POST /api/orgs/[orgId]/task-instances — Task instances with filters
  • GET/POST/DELETE .../task-instances/[id]/assignees — Assignee management
  • PATCH .../task-instances/[id]/status — Update status
